# 我花了两周,把一行链接做成了一个能发给别人的软件

写在前面:这篇不讲怎么用,就是想记一下这两周是怎么过来的。趁还没忘。

我做这个东西的起点很俗。我平时听挺多英文播客,也看技术长视频,但说实话,我要的不是"看完",是"知道它讲了啥,顺手存下来,以后能翻回来"。试过几个现成的工具,要么收费,要么得把我的东西传到它服务器上,要么吐给你一坨没分段的转录,读起来比直接听还累。

那就自己写一个吧。当时想得特别简单:粘个链接进去,有字幕就抓字幕,没字幕就转录,最后让大模型给我整理成干净的文字和摘要。三步,能有多难。

结果这两周,基本每一步都在收拾我。

## 第一天就翻车了

搭完最小可用的转录流程后,我干的第一件事特别小:把默认的 OpenAI 地址,换成我自己在用的国产模型地址。

就这么一下,它崩了。

而且崩的不是我以为的地方。先是代理报错,然后是模型名对不上——代码里写死了一个默认模型,我的服务商根本不认这个名字。两个看起来八竿子打不着的小毛病,反倒让我停下来想了个挺大的问题:这软件到底该"知道"用户多少东西?

我后来想通了,答案是:啥都别知道。密钥也好、用哪个模型也好、连哪个服务也好,全让用户自己在界面里填,只存在他自己浏览器里,我的后端一概不碰、不存,连环境变量都不读。

当时纯粹是图省事,不想搭账号、不想管别人的密钥。但这个决定后来帮了大忙——正因为我这边什么秘密都不留,后面才敢把它打包成软件直接发给别人。挺多事都是这样,你以为是顺手做的,回头看才发现是地基。

## 三小时的视频,把我逼出了一个想法

早期有个体验我自己都受不了。丢一个三小时的采访进去,得干等十几分钟,屏幕上啥也没有,然后"啪"一下全出来了。

我盯着那个空白的进度条想,用户到底最想先看到啥?是摘要啊。逐字稿当然也要,但它可以慢慢来,摘要得先给我。

于是我把流程拆成两条并排跑的路。拿到原始文字之后,一条立刻去生成摘要、好了马上推到屏幕上;另一条在后台慢慢把逐字稿润色干净,弄完了再补上去。你读着摘要的工夫,那边全文也差不多好了。

顺带还搞明白一个一直没想通的事:为啥长文本非得切成小块处理。我一开始以为是模型"读不下"那么长。后来才反应过来,卡的不是读,是写——模型一次能吐出来的字数有上限,你让它润色一篇八万字的稿子,它写到一半就被截断了。想明白这点之后,后面好几处优化才算找着北。

## 一个连字符,改了我大半个晚上

这个 bug 我印象最深,因为它太安静了。

我发现个怪事:明明好多 YouTube 视频是有字幕的,我这工具老说"找字幕失败",然后老老实实去下载音频跑转录,又慢又费电。更邪门的是日志,前一行还写着"发现字幕",下一行就是"字幕获取失败"。发现了又用不上,你说气不气。

查到最后,罪魁是一个字符。代码里某处该写 `yt_dlp`,被写成了 `yt-dlp`,一个下划线写成了横杠。

最坑的是,这玩意儿在 Python 里语法是合法的,它被当成了一个减法,所以文件能正常加载、屁事没有。只有真跑到那一行才会悄悄炸,而炸出来的错又正好被一段"出错就回退去下载音频"的保护逻辑稳稳接住了,一点痕迹都不留。

所以它的表现就是:永远不声不响地走最慢那条路。

修完我特意把这种"出错就默默兜底"的地方都加上了日志。宁可它吵一点,我也不想再被这么骗一回。你以为自己写的是容错,搞不好只是在替 bug 打掩护。

## RSS 这块,我是真没想到会这么麻烦

我想让它能订阅播客和博客,定期自动处理。加个订阅功能,能费多大劲。

第一版我把订阅存在浏览器本地。然后批量导 40 个源,报"32 个失败"。我先怀疑并发太猛,又怀疑是源本身有问题,折腾一圈都不对。最后把日志打详细了才看明白:浏览器本地存储被撑爆了。有的播客一个源就返回上千条节目,几个下去配额就满了。换了个扛得住的存法、每个源只留最新一批,才好。

还碰到个有意思的源。有位挺有名的工程师,他博客我怎么订都说"没有可处理的内容"。查了半天,发现他订阅源里每篇文章的正文是空的,只给标题和链接,正文得去他网页上才有。这不是 bug,人家博客就这么设计的。我最后加了条兜底:正文为空、但有链接的时候,自动去原文网页把内容抓回来再处理。

做这种跟外部世界对接的功能,你面对的根本不是什么标准,是一百种不按套路出牌的现实。

## "干脆打包成软件吧"

某天我突然想到,这东西前后端本来就是一个进程,我干嘛不直接打包成一个双击就能用的软件?让不懂命令行的人也能用上,填个密钥就开转,多好。

我选了用系统自带的浏览器内核来显示界面,没塞那个几百兆的 Chromium 进去。然后把转录引擎、下载工具、连音视频处理器都打进了安装包。第一个 macOS 版本出来那会儿,我还挺得意的。

得意了大概一天。

## 发给别人,才是真正的开始

我把打好的包发给几个朋友试。然后报错跟雪片一样飞回来,清一色是我自己机器上压根不会出现的。

软件起都起不来,因为数据库想往一个只读的地方写文件。我打进去的那个音视频处理器,依赖一堆只有我电脑上才装了的库,到别人那直接找不着。下载 YouTube 报"格式不可用",因为我用的下载工具得联网去 GitHub 拉段脚本破解签名,而人家那边上不去 GitHub。好不容易下下来,转录又崩,因为打包之后有个模型文件的路径对不上了。

每一个,都是"在我这儿好好的"。

这几天我学到的东西比写功能那阵多得多。"在我电脑上能跑"和"在一台啥都没装的陌生电脑上能跑",根本是两码事。后者你什么都不能假设:不能假设有网,不能假设能上 GitHub,不能假设系统装了哪个库,不能假设你有权限往哪写文件。我后来甚至养成个习惯,把自己电脑上的运行环境变量全清空,假装这是台一无所有的机器来跑,能活下来才敢发出去。

还有个特别朴素但特别重要的事:用户根本看不到你的日志。前面那些报错,全是用户从黑框里复制一长串发给我,我才知道出了啥事。所以这阶段我做的最值钱的"功能",其实是把日志老老实实写到文件里这么个东西。用户一点都感觉不到它,但没有它,这软件根本没法维护下去。

## 重写一遍,还有"取消"这个看似简单的事

功能一多,最早那个所有逻辑挤在一个文件里的网页,改起来就越来越要命。我咬牙重写了一遍前端,换成了现在这套组件化的。重写从来不是什么轻松事,但换来的是之后每次改动都顺手不少,值。

后端这边,让我想最久的反而是个听着特简单的需求:点"取消"的时候,任务得真的停下来。

我本来以为这就是删条记录的事。挖下去才发现挺尴尬,我那个"取消"是假的。界面上任务是没了,可机器后台,下载还在跑,CPU 还在为一个早就被"取消"掉的转录满负荷空转。因为这些重活跑在一种 Python 杀不掉的线程里,你以为你停了它,其实你只是不等它了,它本人还在那埋头干。

为了真把它"杀干净",我一度想把整个架构推倒重做,越想越复杂。是被一句话点醒的:先去看看那些专业工具官方自己是怎么做的,别每回一设想就把架构搞得更臃肿。

于是我老老实实去翻了转录引擎、下载工具、音视频处理器各自的文档和源码。翻完松了口气——它们其实各自都留了优雅的中断方式,根本用不着我抡重型武器。我要做的只是用对它们各自的法子,而不是自己硬来。这种感觉挺微妙的:当你觉得一个问题需要很复杂的方案时,多半是你还没找到别人早就铺好的那条路。

## 现在

两周下来,它从一个"换个链接就崩"的小脚本,长成了能处理三十多个平台、能订阅 RSS 自动跟进、四种语言界面、能打包成双击即用的桌面软件,而且在一台啥都没装的陌生电脑上也基本扛得住。

回头看,真正难的几乎都不是"再写个新功能",是那些不起眼的、关于边界的问题。一个软件该不该握着用户的秘密。出错的时候是默默兜住还是大声喊出来。"在我电脑上能跑"到底算不算数。你想"杀干净"一个任务的时候,你真的杀干净了吗。

这些问题没一个写在我最初的计划里,全是被一个个具体的坑,在某个具体的深夜,逼着我想明白的。
