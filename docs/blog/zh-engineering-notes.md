# 做一个转录工具,这几个坑比我想的深得多

上一篇我讲了做 AI Transcriber 这两周的整体经过,偏故事。这篇想补几个技术上的细节,给同样在折腾转录、ASR、大模型后处理的人看。不是教程,就是几个"我以为很简单、结果折腾很久"的地方,以及我最后是怎么想通的。

## 一、能不转录,就别转录

最开始我默认的流程是:下载音频 → 跑 Whisper → 整理。后来才意识到一件事:**大部分 YouTube 视频本身就有字幕**,人工的或者自动生成的。有字幕你还吭哧吭哧下音频跑模型,又慢又费 CPU,纯属给自己找罪受。

但"抓字幕"这事不能莽。如果直接让 yt-dlp 带着字幕一起下,它会把音视频流也拉下来,慢且浪费。我后来改成两段式:**先只探测、不下载任何东西**,确认有字幕、是哪个语言,再单独把字幕文件拉下来。

```python
# 第一步:只探测信息,不下载任何媒体流
info, _ = await self._extract_info_with_cookie_fallback(url, check_opts, 60.0)
manual_subs = info.get("subtitles") or {}
auto_caps   = info.get("automatic_captions") or {}

# 第二步:只下字幕,跳过音视频
dl_opts = self._get_download_opts(url, {
    "writesubtitles":    prefer_manual,
    "writeautomaticsub": not prefer_manual,
    "subtitlesformat":   "vtt/srt/best",
    "skip_download":     True,   # ← 关键:只要字幕
})
```

几个被现实教育出来的细节,光看文档想不到:

- **`live_chat` 必须过滤掉**。直播回放的"字幕"里混着聊天室消息轨道,你不处理,转录稿会变成弹幕大杂烩。
- **手动字幕优先于自动字幕**。自动字幕是机器识别的,质量明显差,有人工的就别用机器的。
- **YouTube 自动字幕有个"滚动追加"格式**,这个最坑。同一句话会被拆成好几段、逐字往上叠(就为了那个字一个个蹦出来的效果),你直接解析会拿到一堆重复的半句。得按组去重,只留每组最后那版完整的。

抓到字幕之后,我把它格式化成**和 Whisper 输出一模一样的结构**。这步看着没什么,其实挺重要:下游做整理、摘要、导出的代码,完全不需要知道这段文字到底来自字幕还是转录,一套逻辑通吃。

这条快路径覆盖了大多数 YouTube 视频。只有真没字幕,才落到下面那条慢路。

## 二、Whisper 的难点不是跑通,是让它别瞎编

`faster-whisper` 跑起来很容易。难的是让它在**长音频、静音段、有噪音的片段**上别胡说八道。Whisper 这类模型有个老毛病:遇到静音或者噪音,它会"幻觉"出一段根本不存在的话,而且一旦错一句,后面会跟着连环重复同一句。

我现在这套参数,基本是被一段段烂转录逼出来的:

```python
segments, info = self.model.transcribe(
    audio_path,
    temperature=[0.0, 0.2, 0.4],      # 低温失败了再用高温重试
    vad_filter=True,                   # 先用 VAD 把非语音段切掉
    vad_parameters={"min_silence_duration_ms": 500, "speech_pad_ms": 400},
    compression_ratio_threshold=2.4,   # 压缩比异常 = 大概率在重复
    condition_on_previous_text=False,  # ← 这条最关键
)
```

最有用的两个决定:

一是 **`condition_on_previous_text=False`**。默认情况下 Whisper 会拿前一段的输出当下一段的上下文,本意是让句子连贯,代价是**错误会累积**——一旦幻觉出一句,它会"参考"这句继续编,滚成连环重复。关掉它,每段独立解码,牺牲一点点连贯性,换来不胡说,我觉得值。反正后面还有大模型整理来补连贯。

二是 **先用 VAD 把非语音段切掉**。静音和纯音乐段是幻觉重灾区,先用语音活动检测把它们摘出去,Whisper 压根不去解码这些段。`speech_pad_ms=400` 是踩出来的,留白太小会把句子的头尾字给切了。

## 三、CPU 上的模型选型,和一个"先能用"的妥协

做桌面软件,我得假设用户没有显卡,所以一切围绕 **CPU + int8 量化**。

默认模型我选了 `large-v3-turbo`:解码比 large-v3 快了好几倍,int8 量化后只占一点几个 G 内存,中英日韩都覆盖。算是 CPU 上精度和速度的甜点。

但它有一点几个 G,**塞不进安装包**。我的折中是:安装包里只内嵌一个一百多兆的小模型(base)保底,turbo 在软件**第一次启动时后台悄悄下载**。下载期间用户来了任务怎么办?**先用 base 顶上**,而不是卡在那等下载完。

```python
def _resolve_available_size(size):
    if size == BUILTIN_MODEL or is_downloaded(size):
        return size
    # turbo 还在后台下,这次先用内嵌的 base,别让任务卡死
    return BUILTIN_MODEL
```

效果就是:用户一打开就能用(base),用着用着 turbo 悄悄下好了,后面的任务自动升级。网络不通就一直用 base,质量降一档但功能不残。这种"先给个能用的,好的东西后台慢慢补上"的思路,做桌面工具比"必须先下完两个 G 才能开张"友好太多。

## 四、"取消"这件事,让我差点把架构推倒重来

这是我折腾最久、也最有意思的一个。需求一句话:点取消,任务得真的停。

我本来以为就是删条记录。挖下去发现尴尬了——我那个"取消"是**假的**。界面上任务没了,机器后台,下载还在跑、CPU 还在为一个早就被取消的转录满负荷空转。因为这些活跑在一种 Python 杀不掉的线程里,你 `cancel()` 掉的只是"对线程的等待",线程本人还在埋头干。

为了真把它杀干净,我一度想把整个流程拆进独立子进程,取消时直接 kill 进程。但这么搞,模型就没法常驻复用了——子进程一死,它内存里那一点几个 G 的模型也没了,下个任务又得重新加载。"杀得干净"和"模型复用"成了死结。

卡在这的时候,我强迫自己先别写,去翻了几个工具官方到底怎么处理取消的。翻完豁然开朗,原来根本不用我抡重型武器:

- **Whisper**:`transcribe()` 返回的其实是个**惰性生成器**,你不迭代它就不解码。所以我只要在 `for segment in segments` 的循环里查一个标志,要取消就 `break` + `close()`,后续解码自然就停了。模型还在内存里、热的,**根本不用杀进程**。唯一的代价是只能在"段与段之间"停,正在解的那一段得先跑完——但一段也就几秒,用户完全无感。
- **yt-dlp 下载**:它自己留了口子,在 `progress_hooks` 里抛个 `DownloadCancelled` 就能干净中断,也不用杀进程。
- **ffmpeg**:这个是真没法在进程内取消,因为它是 yt-dlp 拉起来的**独立子进程**。但它响应信号,我只要用 `start_new_session` 把它放进一个独立进程组,取消时按进程组整组 kill 就行(单杀顶层会漏掉它派生的孙子进程)。

我还顺手去看了那些"重型队列库"(Celery 之类)是不是能白送我这个能力。结论挺反直觉:Celery 之所以能真杀任务,是因为它本来就把每个任务跑在独立子进程里,而且要拖一个 Redis/RabbitMQ。对我这种单用户桌面应用,纯属过度设计。**问题根本不在队列框架,在底层那三个库的活儿是进程内打不断的。** 想通这点,我就老老实实保留了自己手写的轻量队列,只是把取消用"各家官方的法子 + 一处进程组"补对。

这件事给我的体会挺深的:**当你觉得一个问题需要很复杂的方案时,大概率是你还没找到别人早就铺好的那条路。** 克制住"自己造一套"的冲动,先去读上游的设计,往往能让架构收敛,而不是越搞越大。

(顺带一提,翻代码的时候还发现一个潜伏的 bug:Whisper 那个逐段解码的循环,本来是跑在主事件循环线程里的,转录期间会把整个服务的其它请求都卡住。趁这次一起挪进了工作线程。)

## 五、最后一个,是产品决策不是算法:摘要先出来

整理一篇长转录稿要十几秒甚至更久,但用户最想要的其实是摘要。所以我让摘要和全文整理**并行跑**,摘要一好就先通过 SSE 推到屏幕上,全文在后台接着清洗,完了再补上。

同样的总耗时,用户的体感完全不一样——他读着摘要的工夫,全文也差不多好了,而不是对着空白进度条干等。
